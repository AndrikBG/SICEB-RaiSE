**Files:**

**TDD Cycle:**
- RED: Import views, verify routes render, TypeScript compiles

## Execution Order

```
T1 (API clients + branch switching) → T2 (branches) ─┐
                                    → T3 (inventory)  ├→ T5 (integration)
                                    → T4 (tariffs)   ─┘
```

T1 first: API clients and nav integration are foundation for all views. T2/T3/T4 can theoretically run in parallel (independent views) but sequential execution is cleaner for single developer — order by dependency on the most complex API surface. T5 validates everything together.

Note: T3 is the largest task (two views + WebSocket real-time). If it grows too large during implementation, consider splitting InventoryDashboardView and ServiceInventoryView into separate commits.

## Risks

| Risk | Mitigation |
|------|------------|
| Backend DTO shapes may not match API client types | Cross-reference TariffController/BranchController response records (already done in design) |
| WebSocket subscription may fail silently if token is stale after branch switch | Disconnect/reconnect pattern picks up new token via `beforeConnect` hook |
| `inventory:read_service` users need a serviceId to filter by — where does it come from? | Check if `user.staffId` maps to service assignment, or if a separate endpoint is needed |
| Frontend build may fail on strict TypeScript with existing code | Run `tsc --noEmit` after T1 to catch early |
