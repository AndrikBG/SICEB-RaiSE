# Story: Scalability Validation

## User Story
As a system architect,
I want to validate that the multi-branch infrastructure scales from 3 to 15 branches with less than 10% query degradation,
so that SICEB can onboard new branches with confidence in system performance.

## Acceptance Criteria

### Scenario: Query latency scales linearly with branch count
```gherkin
Given the system has data seeded for 3 branches
  And baseline query latencies are recorded
When the system is scaled to 15 branches with equivalent data
Then per-branch queries (partition-pruned) degrade less than 10%
  And cross-branch admin queries degrade less than 10%
```

### Scenario: WebSocket sustains concurrent connections
```gherkin
Given 15 branches with ~10 concurrent users each
When all 150 users establish WebSocket connections
Then all connections remain stable for sustained period
  And message delivery latency remains under 2 seconds
```

### Scenario: RLS with partitioning scales linearly
```gherkin
Given inventory_deltas and stock_current are partitioned by branch_id
  And RLS enforces branch isolation
When per-branch queries execute across 3, 9, and 15 branches
Then EXPLAIN ANALYZE shows partition pruning active
  And query time scales linearly (not exponentially)
```

### Scenario: Patient search sub-1s at volume
```gherkin
Given the patient table has >50,000 records
When a search by name or identifier is executed
Then results return in under 1 second
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| 3 branches, 1K deltas each | Benchmark queries | Baseline latencies recorded |
| 15 branches, 1K deltas each | Benchmark queries | <10% degradation vs baseline |
| 150 WebSocket connections | Sustained load | All connections stable, <2s delivery |
| 50K patient records | Search by name | <1s response time |
