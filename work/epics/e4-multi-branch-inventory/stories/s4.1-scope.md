# Scope — S4.1: Branch Management and Context Switching

## In Scope

- `BranchRegistrationService`: register (unique name, address) + deactivate (preserve history)
- `BranchOnboardingOrchestrator`: 5-step idempotent workflow with progress tracking
  - Step 1: Create DB partitions for inventory_items and inventory_deltas
  - Step 2: Seed default service catalog from org template
  - Step 3: Copy active tariff catalog to new branch
  - Step 4: Initialize empty inventory items per service
  - Step 5: Mark onboarding_complete = true
- `BranchContextService`: mid-session switch via JWT re-issuance (ESC-03 <3s)
- `BranchConfigurationStore`: persistence layer
- Flyway migrations: branch_onboarding_status, branch_service_catalog, branches.onboarding_complete
- Permission seed: `branch:manage`, `branch:read`
- REST endpoints: POST/GET /branches, POST /branches/:id/deactivate, POST /session/branch, GET /branches/:id/onboarding
- RLS policies on new tables
- `Branch` entity: add fields (phone, email, opening_time, closing_time, branch_code) + mutability
- Flyway migration: alter branches table with new columns
- Move `BranchRepository` from `platform.iam.repository` to `platform.branch.repository`
- "Remember last branch" preference per user (last_active_branch_id)
- Audit events for branch registration, deactivation, and context switch

## Out of Scope

- PWA branch management views (S4.4)
- Inventory tables and partitions creation logic (referenced by onboarding but actual schema is S4.2)
- Tariff table (S4.3)
- WebSocket configuration changes (S4.2)

## Done Criteria

- [ ] Branch registered via POST /api/branches with unique name validation
- [ ] Branch deactivated via POST /api/branches/:id/deactivate, preserves data
- [ ] 5-step onboarding orchestrator completes, each step idempotent and resumable
- [ ] Onboarding total time < 1 hour (ESC-01)
- [ ] Branch context switch via POST /api/session/branch returns new JWT in < 3 seconds (ESC-03)
- [ ] Unassigned branch switch rejected with 403
- [ ] RLS enforced on branch_onboarding_status and branch_service_catalog
- [ ] Permissions `branch:manage` and `branch:read` seeded and linked to system roles
- [ ] BranchRepository relocated to platform.branch package
- [ ] Audit events emitted for registration and deactivation
- [ ] All tests pass (unit + ArchUnit)
