### T2 — BranchRegistrationService + BranchOnboardingOrchestrator (M)

**Goal:** Branch lifecycle management with automated onboarding.

**Files:**
- **Create:** `apps/backend/src/main/java/com/siceb/platform/branch/service/BranchRegistrationService.java`
  - `registerBranch(RegisterBranchCommand)` — unique name validation, create Branch, trigger onboarding, emit audit event
  - `updateBranch(UUID branchId, UpdateBranchCommand)` — edit name/address/phone/email/hours
  - `deactivateBranch(UUID branchId)` — set isActive=false, emit audit event
- **Create:** `apps/backend/src/main/java/com/siceb/platform/branch/service/BranchOnboardingOrchestrator.java`
  - `startOnboarding(UUID branchId)` — initializes 5 step records in PENDING
  - `runOnboarding(UUID branchId)` — executes pending/failed steps in order, each idempotent
  - Step 1 `PARTITION_SETUP`: stub (marks COMPLETED — actual DDL in S4.2)
  - Step 2 `SERVICE_CATALOG`: seeds from org template
  - Step 3 `TARIFF_COPY`: stub (tariff table in S4.3)
  - Step 4 `INVENTORY_INIT`: stub (inventory tables in S4.2)
  - Step 5 `FINALIZE`: sets `branch.onboardingComplete = true`
- **Create:** `apps/backend/src/main/java/com/siceb/platform/branch/command/RegisterBranchCommand.java` — record with validation
- **Create:** `apps/backend/src/main/java/com/siceb/platform/branch/command/UpdateBranchCommand.java` — record
- **Create:** `apps/backend/src/main/java/com/siceb/platform/branch/exception/BranchException.java` — extends RuntimeException with ErrorCode

**TDD:**
- RED: `registerBranch` with valid data → branch created + onboarding started; duplicate name → 409; `deactivateBranch` → isActive=false; onboarding resumes from failed step
- GREEN: Implement services
- REFACTOR: Extract common patterns if any

**Verification:**
- Unit tests: register (valid, duplicate), deactivate (active, inactive, not found), onboarding (full run, resume after failure, idempotent re-run)
- ArchUnit passes

**AC:** AC1 (register with unique name), AC2 (onboarding idempotent)
**Dependencies:** T1

### T3 — BranchContextService + AuthenticationService Integration (S)

**Goal:** Mid-session branch switching without re-authentication.

**Files:**
- **Create:** `apps/backend/src/main/java/com/siceb/platform/branch/service/BranchContextService.java`
  - `switchBranch(UUID userId, UUID targetBranchId)` → validates assignment + active status → loads User + MedicalStaff → calls `JwtTokenService.generateAccessToken(user, targetBranchId, medicalStaff)` → saves `lastActiveBranchId` → emits audit event → returns `BranchSwitchResult(accessToken, branchId, branchName)`
  - `resolveInitialBranch(User user)` → returns `lastActiveBranchId` if still assigned, else first assigned branch, else error
- **Modify:** `apps/backend/src/main/java/com/siceb/platform/iam/service/AuthenticationService.java`
  - `login()`: replace hardcoded `user.getBranchId()` with `branchContextService.resolveInitialBranch(user)` for default branch selection
  - `refresh()`: use `user.getLastActiveBranchId()` or current JWT branch

**TDD:**
- RED: `switchBranch` with assigned branch → new JWT with correct activeBranchId; unassigned → 403; deactivated → 403; `resolveInitialBranch` returns last active, falls back to first assigned
- GREEN: Implement service, wire into AuthenticationService
- REFACTOR: Clean up AuthenticationService branch resolution

**Verification:**
- Unit tests: switch (valid, unassigned, deactivated, non-existent), resolve initial (last active, fallback, no branches)
- Token contains correct `activeBranchId` claim after switch

**AC:** AC3 (switch < 3 seconds), AC4 (unassigned/deactivated → 403)
**Dependencies:** T1, T2 (needs BranchRepository with new methods)
**Size:** S

## Execution Order

```
T1 (Migrations + Entity) ─── foundational, no deps
    │
    ├──→ T2 (Registration + Onboarding) ─── core business logic
    │       │
    │       └──→ T3 (Context Switch) ─── depends on BranchRepository methods from T2
    │               │
    │               └──→ T4 (Controller) ─── HTTP surface, depends on all services
    │                       │
    │                       └──→ T5 (Integration) ─── validates everything
```

**Rationale:**
- T1 first: everything depends on schema and entities
- T2 before T3: context switch needs branch registration to create branches to switch to
- T3 before T4: controller wires both services
- T5 last: integration test validates the whole stack

No parallel opportunities — each task builds on the previous in this story.
