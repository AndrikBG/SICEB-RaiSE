## What to Improve

- **T4 controller test gap**: `@RequestAttribute("userId")` was never set by any filter, but the unit test passed because it called the method directly. A mock MVC test would have caught this in T4 instead of T5.
- **ErrorCode exhaustiveness**: the `default -> 500` in GlobalExceptionHandler silently swallowed `CONFLICT`. A compile-time exhaustive switch (no default) would force explicit handling.
- **Transaction boundary awareness in audit**: `REQUIRES_NEW` + FK to uncommitted entity is a predictable failure pattern. Should be a design-time checklist item for any `REQUIRES_NEW` usage.

## Heutagogical Checkpoint

### 1. What did I learn?
`Propagation.REQUIRES_NEW` creates a visibility trap: the new transaction cannot see uncommitted rows from the outer transaction. When audit events reference a newly created entity via FK, the audit insert fails. Fix: use the caller's existing context (TenantContext) for audit FK, put new entity ID in non-FK fields.

### 2. What would I change about the process?
Add a mock MVC integration test for at least one endpoint per controller. Direct method invocation tests verify business logic wiring but not HTTP input resolution (SecurityContext, request attributes, headers).

### 3. Are there improvements for the framework?
Consider a guardrail that flags `Propagation.REQUIRES_NEW` methods that receive entity IDs from the outer transaction — these are FK violation candidates. Also, exhaustive switch statements on ErrorCode (no default) as a linting rule.

### 4. What am I more capable of now?
I can reason about Spring transaction boundary visibility in multi-layer architectures (service -> audit -> DB). I also have a concrete catalog of which bugs unit tests catch vs. which require integration testing against a real DB.
