# Design — S4.1: Branch Management and Context Switching

> Complexity: Moderate (10 components, established patterns, novel orchestrator)
> US: [US-071](../../../../docs/Requirements/US/US-071_register_branch.md), [US-074](../../../../docs/Requirements/US/US-074_select_active_branch.md)

## Problem & Value

The `Branch` entity exists as a read-only record seeded in V002. There is no way to register, edit, deactivate branches, or switch branch context mid-session. This blocks multi-branch operations — the clinic network cannot onboard new locations or have staff work across branches.

After this story: administrators can register and manage branches with automated onboarding, and users can switch branches in <3 seconds without re-authentication.

## Approach

Three services in `platform.branch`, one new REST controller, Flyway migrations for schema changes + permission seeds, and a modification to `AuthenticationService` for JWT re-issuance.

**Onboarding orchestrator pattern:** Choreographed 5-step workflow with progress table. Each step is an independent method that checks its completion status before executing — idempotent and resumable. No external orchestrator or message broker. Steps 1 and 4 reference inventory tables that don't exist yet (S4.2) — those steps will execute DDL for partitions but actual table creation is in S4.2 migrations. In S4.1 they are stubs that mark themselves complete.

**Context switch pattern:** `BranchContextService.switchBranch()` validates assignment → loads User + MedicalStaff → calls `JwtTokenService.generateAccessToken(user, newBranchId, medicalStaff)` → returns new JWT. The caller (AuthController or BranchController) sets the HttpOnly cookie. No re-authentication needed — the user is already authenticated, we just re-issue with different `activeBranchId` claim.

**Missing entity strategy (PAT-E-523):** Deactivate non-existent branch → 404. Switch to unassigned branch → 403. Register with duplicate name → 409. All rejected with `ErrorResponse`.

## Examples

### Register Branch

```
POST /api/branches
Authorization: Bearer <jwt with branch:manage>
Content-Type: application/json

{
  "name": "Sucursal Norte",
  "address": "Av. Universidad 1500, Monterrey, NL 64000",
  "phone": "+528112345678",
  "email": "norte@clinica.mx",
  "openingTime": "08:00",
  "closingTime": "20:00",
  "branchCode": "NTE"
}

→ 201 Created
{
  "branchId": "01970f3a-...",
  "name": "Sucursal Norte",
  "onboardingStatus": "IN_PROGRESS"
}
```

Error — duplicate name:
```
→ 409 Conflict
{
  "code": "SICEB-4001",
  "message": "A branch with this name already exists",
  "correlationId": "..."
}
```

### Switch Branch Context

```
POST /api/session/branch
Authorization: Bearer <jwt with activeBranchId=AAA>
Content-Type: application/json

{
  "branchId": "01970f3b-..."
}

→ 200 OK
Set-Cookie: refresh_token=<new>; HttpOnly; Secure; SameSite=Strict
{
  "accessToken": "<new JWT with activeBranchId=01970f3b-...>",
  "branchId": "01970f3b-...",
  "branchName": "Sucursal Centro"
}
```

Error — not assigned:
```
→ 403 Forbidden
{
  "code": "SICEB-2002",
  "message": "User is not assigned to requested branch",
  "correlationId": "..."
}
```

### Deactivate Branch

```
POST /api/branches/01970f3a-.../deactivate
Authorization: Bearer <jwt with branch:manage>

→ 200 OK
{
  "branchId": "01970f3a-...",
  "isActive": false
}
```

### List Branches

```
GET /api/branches?includeInactive=true
Authorization: Bearer <jwt with branch:read>

→ 200 OK
{
  "branches": [
    { "branchId": "...", "name": "Sucursal Norte", "isActive": true, "onboardingComplete": true },
    { "branchId": "...", "name": "Sucursal Centro", "isActive": true, "onboardingComplete": true },
    { "branchId": "...", "name": "Sucursal Sur", "isActive": false, "onboardingComplete": true }

```

Non-admin users only see their assigned branches (filtered by `user.branchAssignments`).

## Database Schema Changes

### V015 — Branch Schema Extension

```sql
-- Extend branches table
ALTER TABLE branches ADD COLUMN phone VARCHAR(20);
ALTER TABLE branches ADD COLUMN email VARCHAR(200);
ALTER TABLE branches ADD COLUMN opening_time TIME;
ALTER TABLE branches ADD COLUMN closing_time TIME;
ALTER TABLE branches ADD COLUMN branch_code VARCHAR(20);
ALTER TABLE branches ADD COLUMN onboarding_complete BOOLEAN NOT NULL DEFAULT false;

-- Add last_active_branch_id to users
ALTER TABLE users ADD COLUMN last_active_branch_id UUID REFERENCES branches(id);

-- Onboarding progress tracking
CREATE TABLE branch_onboarding_status (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    branch_id   UUID NOT NULL REFERENCES branches(id),
    step_name   VARCHAR(50) NOT NULL,
    status      VARCHAR(20) NOT NULL DEFAULT 'PENDING',
    completed_at TIMESTAMPTZ,
    error_message TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_onboarding_step UNIQUE (branch_id, step_name),
    CONSTRAINT chk_onboarding_status CHECK (status IN ('PENDING', 'IN_PROGRESS', 'COMPLETED', 'FAILED'))
);
CREATE INDEX ix_onboarding_branch ON branch_onboarding_status(branch_id);

-- Service catalog template
CREATE TABLE branch_service_catalog (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    branch_id   UUID NOT NULL REFERENCES branches(id),
    service_name VARCHAR(200) NOT NULL,
    service_code VARCHAR(50),
    is_active   BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_service_branch UNIQUE (branch_id, service_name)
);
CREATE INDEX ix_service_catalog_branch ON branch_service_catalog(branch_id);
```

## Decisions

| # | Choice | Rationale | Alternatives |
|---|--------|-----------|-------------|
| D-044 | Onboarding steps 1 and 4 are stubs in S4.1 (inventory tables don't exist yet) | S4.2 creates inventory schema; orchestrator becomes fully functional then | Block S4.1 on S4.2 (circular), create empty partitioned tables now (premature) |
| D-045 | `last_active_branch_id` on users table (not separate preferences table) | Single column, simple query, matches US-074 "remember last branch" | Separate key-value preferences table (overengineered for one field) |
| D-046 | Branch context switch returns new access token in response body + new refresh token in HttpOnly cookie | Consistent with existing auth flow; PWA stores access token in memory | Return only cookie (PWA can't read HttpOnly), return only body (loses refresh rotation) |

## Acceptance Criteria

> Full Gherkin in US-071 and US-074. MUST criteria for this story:

- **AC1:** `POST /api/branches` creates branch with unique name validation → 201 or 409
- **AC2:** 5-step onboarding runs automatically, each step idempotent, total < 1 hour
- **AC3:** `POST /api/session/branch` returns new JWT with updated `activeBranchId` in < 3 seconds
- **AC4:** Unassigned branch switch → 403; deactivated branch switch → 403
- **AC5:** `branch:manage` and `branch:read` permissions seeded, linked to system roles
- **AC6:** RLS enforced on `branch_onboarding_status` and `branch_service_catalog`
- **AC7:** Audit events for registration, deactivation, and context switch

**SHOULD:**
- Branch edit (update name/address/phone/email/hours)
- `GET /api/branches` filtered by admin vs non-admin
- `last_active_branch_id` saved and used as default on next login

**MUST NOT:**
- Create inventory tables or partitions (S4.2)
- Modify WebSocket configuration (S4.2)
- Create PWA views (S4.4)
