# Design — S4.3: Tariff Configuration

## Problem & Value

**Problem:** No tariff management exists — services have no pricing, blocking billing (Phase 5) and the tariff UI (S4.4).

**Value:** Enables temporal pricing with immutable history. Price changes take effect on a future date without altering past charges. Completes M2 (Core Backend).

## Complexity

Simple-to-Moderate: 4 components (migration, entity, service, controller), proven CRUD + temporal query pattern, no novel algorithms.

## Approach

Standard Spring Data JPA service with temporal effective-date resolution. The `service_tariffs` table stores one row per price version; the "active" tariff is resolved via `MAX(effective_from) WHERE effective_from <= NOW()`. Updates never mutate existing rows — they insert a new row with a future `effective_from`. This follows D-042.

**Data mutation strategy:** CreateTariff with non-existent `serviceId` or `branchId` → reject with 404 (FK constraint + application validation). Duplicate `(service_id, branch_id, effective_from)` → reject with 409 Conflict.

**Package:** `com.siceb.domain.billing` — the `package-info.java` stub already exists (IC-01 from Phase 1). Tariff is the first concrete billing component; placing it here is consistent with the module architecture.

## Examples

### API: Create Tariff

```http
POST /api/tariffs
Authorization: Bearer <jwt with tariff:manage>

{
  "serviceId": "a1b2c3d4-...",
  "basePrice": 150.0000,
  "effectiveFrom": "2026-05-01T00:00:00Z"
}

→ 201 Created
{
  "tariffId": "f5e6d7c8-...",
  "serviceId": "a1b2c3d4-...",
  "branchId": "b1c2d3e4-...",
  "basePrice": 150.0000,
  "effectiveFrom": "2026-05-01T00:00:00Z",
  "createdBy": "u1v2w3x4-...",
  "createdAt": "2026-04-13T10:00:00Z"
}
```

### API: Update Tariff (new effective-date entry)

```http

```

### API: Resolve Active Tariff

```http
GET /api/tariffs/active?serviceId=a1b2c3d4-...
Authorization: Bearer <jwt with tariff:read or tariff:manage>

→ 200 OK
{
  "tariffId": "f5e6d7c8-...",
  "basePrice": 150.0000,

  ...

```

### API: List Tariffs (with history)

```http
GET /api/tariffs?serviceId=a1b2c3d4-...&includeHistorical=true&page=0&size=20

→ 200 OK (paginated, ordered by effectiveFrom DESC)
```

### API: Search Tariffs by service name

```http
GET /api/tariffs/search?query=Consulta&page=0&size=20

→ 200 OK (tariffs whose service name contains "Consulta")
```

### Error: Negative price

```http
POST /api/tariffs  { "basePrice": -10.00, ... }
→ 400 Bad Request  { "code": "SICEB-1001", "message": "Base price must be non-negative" }
```

### Error: Duplicate effective date

```http
POST /api/tariffs  { same service + branch + effectiveFrom as existing }
→ 409 Conflict  { "code": "SICEB-4001", "message": "Tariff already exists for this service, branch, and effective date" }
```

### Error: Past effective date on update

```http
PUT /api/tariffs/...  { "effectiveFrom": "2025-01-01T00:00:00Z" }
→ 400 Bad Request  { "code": "SICEB-1001", "message": "Effective date must be current or future" }
```

### Data Model: ServiceTariff entity

```java
@Entity
@Table(name = "service_tariffs")
public class ServiceTariff {
    @Id UUID tariffId;
    UUID serviceId;       // FK to branch_service_catalog
    UUID branchId;        // FK to branches, RLS column
    BigDecimal basePrice; // DECIMAL(19,4)
    Instant effectiveFrom;
    UUID createdBy;
    Instant createdAt;
}
```

### SQL: Active tariff resolution

```sql
SELECT * FROM service_tariffs
WHERE service_id = :serviceId
  AND branch_id = :branchId
  AND effective_from <= NOW()
ORDER BY effective_from DESC
LIMIT 1;
```

## Acceptance Criteria

### MUST

1. `service_tariffs` table with RLS, UNIQUE(service_id, branch_id, effective_from), admin_reporting grants
2. `tariff:read` permission seeded (tariff:manage already exists in V010) and granted to appropriate roles
3. CreateTariff validates non-negative DECIMAL(19,4) price, rejects past effective dates
4. Active tariff resolved as MAX(effective_from) <= NOW() — verified with multi-row temporal data
5. UpdateTariff creates new row with future effective_from (original immutable)

### SHOULD

1. SearchTariffs by service name substring (JOIN to branch_service_catalog)
2. ListTariffs supports includeHistorical filter (default: active only)

### MUST NOT

1. Never mutate an existing tariff row — immutable history
2. Never allow cross-branch tariff access via API (RLS + application check)
3. Never accept negative prices

## Decisions

| # | Choice | Rationale | Alternatives |
|---|--------|-----------|-------------|
| D1 | Place in `domain.billing` | Package stub exists (IC-01), tariff is billing's first concrete component | `domain.tariff` (unnecessary new package) |
| D2 | `branchId` from JWT principal, not request body | Consistent with InventoryController pattern; RLS alignment | Accept branchId in body (cross-branch risk) |
| D3 | No partitioning on `service_tariffs` | Low data volume (~services x branches x versions); not worth partition overhead | Partition by branch_id like inventory |
| D4 | Search via JOIN to `branch_service_catalog` | Service name lives in catalog; avoid denormalization | Denormalize service_name into tariff |
