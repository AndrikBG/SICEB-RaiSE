# Design — S4.5: Scalability Validation

## Problem & Value

SICEB targets 15 branches but all development validated at 3. Without empirical evidence that partitioning, RLS, and WebSocket infrastructure scale linearly, branch onboarding carries unknown performance risk. This story produces a scalability report — a go/no-go artifact for production rollout.

## Approach

Two test harnesses against Docker Compose (real PostgreSQL, real Spring Boot, real WebSocket):

1. **k6 load scripts** — HTTP and WebSocket scenarios at 3, 9, 15 branch counts
2. **JUnit integration test** — EXPLAIN ANALYZE validation that partition pruning is active

Data seeded via SQL script parameterized by branch count. Report generated from k6 JSON output + EXPLAIN results.

## Key Decisions

### D-045: k6 over JMeter/Gatling
- k6: JavaScript-based, native WebSocket support, CLI-first, lightweight
- JMeter: XML config, heavy UI, overkill for validation
- Gatling: Scala DSL, heavier dependency chain
- k6 runs as standalone binary — no build system integration needed

### D-046: Docker Compose over Testcontainers for load tests
- Load tests need realistic connection pooling, thread scheduling, WebSocket broker
- Testcontainers is great for unit-scope DB validation (used for EXPLAIN ANALYZE)
- Docker Compose gives full stack: PostgreSQL + Spring Boot + WebSocket endpoint

### D-047: SQL seeding over programmatic data generation
- SQL is deterministic, version-controlled, inspectable
- Parameterized by branch count — same script for 3, 9, 15
- Uses `create_inventory_partitions(UUID)` function from V016 migration

## Examples

### k6 HTTP scenario (per-branch query)
```javascript
import http from 'k6/http';
import { check } from 'k6';

const BASE_URL = __ENV.BASE_URL || 'http://localhost:8080';

export default function () {
  const branchId = randomBranch();  // from seeded data
  const res = http.get(`${BASE_URL}/inventory/items?branchId=${branchId}`, {
    headers: { Authorization: `Bearer ${getToken(branchId)}` },
  });
  check(res, {
    'status 200': (r) => r.status === 200,
    'latency < 500ms': (r) => r.timings.duration < 500,
  });
}
```

### k6 WebSocket scenario (150 connections)
```javascript
import ws from 'k6/ws';

export default function () {
  const url = `ws://localhost:8080/ws`;
  const res = ws.connect(url, { headers: { Authorization: `Bearer ${token}` } }, function (socket) {
    socket.on('open', () => {
      // STOMP CONNECT + SUBSCRIBE to branch topic
      socket.send(stompConnect(token));
      socket.send(stompSubscribe(`/topic/branch/${branchId}/inventory`));
    });
    socket.on('message', (msg) => { /* track latency */ });
    socket.setTimeout(() => socket.close(), 30000);
  });
}
```

### EXPLAIN ANALYZE (JUnit)
```java
@Test
void partitionPruningActive() {
    String plan = jdbc.queryForObject(
        "EXPLAIN ANALYZE SELECT * FROM inventory_items WHERE branch_id = ?",
        String.class, testBranchId);
    assertThat(plan).containsIgnoringCase("Seq Scan on inventory_items_");
    // Partition-specific scan, NOT full table scan
}
```

### Seeding SQL (parameterized)
```sql
-- For each branch: create partition, seed 1K inventory items, 5K deltas
DO $$
DECLARE
  b_id UUID;
  branch_ids UUID[] := ARRAY[gen_random_uuid(), ...];  -- N branches
BEGIN
  FOREACH b_id IN ARRAY branch_ids LOOP
    INSERT INTO branches (id, name, ...) VALUES (b_id, 'Branch-' || b_id, ...);
    PERFORM create_inventory_partitions(b_id);
    -- Seed inventory_items and deltas
  END LOOP;
END $$;
```

## Acceptance Criteria

### MUST
- AC1: Data seeding produces consistent test data for 3, 9, and 15 branches
- AC2: Per-branch query p95 degrades <10% from 3→15 branches
- AC3: Cross-branch admin query p95 degrades <10% from 3→15 branches
- AC4: 150 concurrent WebSocket connections sustained with message delivery <2s
- AC5: EXPLAIN ANALYZE confirms partition pruning on inventory_items and inventory_deltas
- AC6: Patient search <1s with >50K seeded records
- AC7: Scalability report committed with baselines, measurements, pass/fail

### SHOULD
- S1: k6 scripts reusable for future scalability regression checks
- S2: Seeding script parameterized (easy to adjust branch count)

### MUST NOT
- MN1: No new production code — validation only
- MN2: No changes to existing migrations or schema
