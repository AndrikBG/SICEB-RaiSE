### T2 — Inventory Database Schema + Materialization Trigger (M)

**Goal:** Partitioned tables, AFTER INSERT trigger with pg_notify, permissions seed, RLS policies. This is the riskiest task — validates R1, R2, R3, R5 from design.

**Files:**
- **Create:** `apps/backend/src/main/resources/db/migration/V016__create_inventory_schema.sql` — `inventory_items` (partitioned), `inventory_deltas` (partitioned), `materialize_inventory_delta()` trigger function, `prevent_delta_update()` immutability trigger, partition creation function, permissions seed (`inventory:read_all`, `inventory:read_service`, `inventory:adjust`), role-permission mappings, admin_reporting grants
- **Modify:** `apps/backend/src/main/resources/db/migration/R__rls_policies.sql` — add `inventory_items`, `inventory_deltas`
- **Create:** `apps/backend/src/test/java/com/siceb/domain/inventory/InventorySchemaIntegrationTest.java` — tests against Testcontainers

**TDD:**
- RED: Integration tests — (1) insert delta → verify `current_stock` updated, (2) insert INCREMENT 50 + DECREMENT 200 → verify exception, (3) insert delta → verify `pg_notify` payload received via JDBC `LISTEN`, (4) attempt UPDATE on `inventory_deltas` → verify blocked, (5) verify partition pruning via `EXPLAIN ANALYZE`, (6) verify RLS blocks cross-branch reads
- GREEN: Write V016 migration with all DDL + trigger + permissions. Update R__rls_policies.
- REFACTOR: Validate pg_notify payload < 8KB. Verify admin_reporting bypasses RLS.

**Verification:** `mvn test -pl apps/backend -Dtest=InventorySchemaIntegrationTest`

**AC:** AC3 (transactional materialization), AC5 (RLS), AC6 (partition pruning), AC9 (append-only)

### T7 — Integration Validation + Docker Compose (S)

**Goal:** End-to-end validation with running software. Confirm all AC, document any gaps.

**Steps:**
1. Start Docker Compose (`docker compose up`)
2. Register a branch → onboarding creates partitions
3. Create inventory items via direct SQL or API
4. POST /inventory/increments → verify 201, stock updated
5. POST /inventory/decrements (insufficient) → verify 409
6. POST /inventory/increments with duplicate key → verify 200 idempotent
7. Verify WebSocket receives push < 2 seconds (browser dev tools)
8. Login as service manager → verify sees only own service items
9. Login as admin → verify cross-branch inventory query
10. Run `EXPLAIN ANALYZE` on single-branch query → verify partition pruning
11. Verify inventory_deltas UPDATE blocked (psql)
12. Subscribe to wrong branch topic → verify rejection

**Verification:** All 10 AC pass. Document results in commit message.

**AC:** AC1–AC10 (all)
**Dependencies:** T1–T6
**Size:** S

## Execution Order

```
T1 (Testcontainers)
 └──→ T2 (Schema + Trigger)
       └──→ T3 (Model + Handler)
             ├──→ T4 (Query + Controller)
             └──→ T5 (WebSocket + Publisher)
                   └──→ T6 (PWA Realtime)
                         └──→ T7 (Integration Validation)
```

**Rationale:** Risk-first. T1→T2 validates the riskiest unknowns (partitioning, triggers, pg_notify) before any Java code. T4 and T5 can be parallelized (no shared files) but will run sequentially for single developer. T6 depends on T5 for auth flow. T7 validates everything.

## Risks

| # | Risk | Task | Mitigation |
|---|------|------|------------|
| R1 | Testcontainers Docker not available on dev machine | T1 | Docker Desktop required; fail fast with clear error |
| R2 | Partition DDL needs superuser | T2 | Testcontainers PG runs as superuser; document GRANT for prod |
| R3 | pg_notify payload > 8KB | T2 | Stripped to 6 fields (~200 bytes); validated in integration test |
| R4 | RealtimeEventPublisher polling latency | T5 | Use 100ms polling interval; measure in T7 |
| R5 | Excel export dependency (Apache POI) | T4 | Check if already in pom; add if needed |
