# Retrospective — S4.4: PWA Views

**Story:** S4.4 | **Size:** M | **Date:** 2026-04-13

## Summary

Delivered 4 PWA views (BranchManagement, InventoryDashboard, ServiceInventory, TariffConfiguration) consuming all E4 backend APIs (S4.1+S4.2+S4.3). Enhanced branch switching with full JWT lifecycle. Estimated 85 min, actual ~50 min (1.7x velocity).

## What Went Well

- **Pattern reuse:** All views followed the established UserManagementView pattern (useState + useEffect + direct API calls). Zero architectural decisions during implementation — all grounded in design.
- **API client typing:** Cross-referencing Spring Boot controller response records to derive TypeScript interfaces caught potential mismatches at design time, not runtime.
- **Branch switching:** The 5-step atomic operation (API → JWT → cache → WS → resubscribe) was fully designed before coding — no surprises during T1.
- **Velocity:** 1.7x faster than estimated. Design-first paid off again — no rework, no blockers.

## What to Improve

- **No browser testing:** TypeScript + ESLint + build verify code correctness but not feature correctness. Views need manual verification with Docker Compose. This is the M3 milestone gap.
- **ServiceInventoryView assumption:** Used `user.staffId` as serviceId filter without verifying the backend mapping. Could be wrong — needs E2E validation.
- **Frontend test coverage:** Zero React component tests. Not a blocker for this story, but growing tech debt. Consider Vitest + Testing Library setup in a future story.
- **TDD cycle for frontend:** Plan's "RED: TypeScript compilation" isn't real TDD. Framework should acknowledge frontend stories have a different verification model.

## Heutagogical Checkpoint

1. **What did I learn?** The existing BranchSelector was a "fake" switch (zustand only, no JWT refresh). Real multi-tenant switching requires the full 5-step atomic operation. This contract gap is invisible to TypeScript — only browser/E2E testing catches it.

2. **What would I change?** Validate backend field mappings (like staffId → serviceId) during design, not defer to E2E. Add a design checklist item: "For each API client field, verify the backend source."

3. **Framework improvements?** Frontend-only stories need a different TDD cycle description in plan templates. "TypeScript clean" is a verification gate, not a test specification.

4. **What am I more capable of now?** Full-stack integration: deriving typed API clients from Spring Boot DTOs, wiring WebSocket with auth-aware resubscription, and the complete branch-switching lifecycle.

## Patterns
