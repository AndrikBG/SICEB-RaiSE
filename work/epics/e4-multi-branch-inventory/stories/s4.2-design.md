# Design — S4.2: Inventory CQRS + Real-Time Propagation

## Complexity: Complex

10+ components, PG triggers + pg_notify, WebSocket security, partitioning, RLS, CQRS command/query split. Full design required.

## Risk Assessment

| # | Risk | Mitigation |
|---|------|------------|
| R1 | PG trigger + pg_notify can't be tested with mocks | Testcontainers infra as T1 — blocks everything else |
| R2 | Partition DDL requires elevated privileges | Test with Testcontainers; document GRANT for production |
| R3 | pg_notify 8KB payload limit | Strip payload to essential fields; validate in integration test |
| R4 | WebSocket JWT auth on STOMP CONNECT is Spring-specific | Existing `ws-client.ts` already sends Bearer token in CONNECT headers |
| R5 | Concurrency on stock materialization | `SELECT ... FOR UPDATE` on `inventory_items` row serializes deltas per item |

## Problem

SICEB has no inventory tracking. Branch managers need to record stock movements (deliveries, usage, adjustments) and see real-time stock levels. Admins need cross-branch visibility.

## Value

Enables clinical operations that depend on knowing what's available (pharmacy dispensation, supply requests). Real-time propagation eliminates manual refresh and stale-data decisions.

## Approach

**CQRS with append-only delta store** — same pattern as `ClinicalEvent` (Phase 2). Write side appends immutable deltas with idempotency keys. PG `AFTER INSERT` trigger materializes stock into `inventory_items.current_stock` and fires `pg_notify`. Read side queries materialized state. WebSocket bridge listens to `pg_notify` and publishes to tenant-scoped STOMP topics.

Key difference from ClinicalEvent: materialization happens in a PG trigger (same transaction), not in Java application code. This is because stock must be consistent immediately — no eventual consistency gap.

## Data Mutation Strategy (PAT-E-523)

- **Missing item on delta command:** Reject with 404 (`RESOURCE_NOT_FOUND`). No silent drops.
- **Negative stock on DECREMENT:** Reject with 409 (`INSUFFICIENT_STOCK`). Trigger raises exception, caught by handler.
- **Duplicate idempotency key:** Return 200 with original result (idempotent). No error, no second write.
- **Item in deactivated branch:** Reject with 403 (`BRANCH_ACCESS_DENIED`). RLS blocks at DB level.

## Database Schema

```sql
-- Parent tables (partitioned by branch_id)
CREATE TABLE inventory_items (
    item_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    branch_id       UUID NOT NULL REFERENCES branches(id),
    sku             VARCHAR(50) NOT NULL,
    name            VARCHAR(200) NOT NULL,
    category        VARCHAR(50) NOT NULL,
    service_id      UUID NOT NULL REFERENCES branch_service_catalog(id),
    current_stock   INTEGER NOT NULL DEFAULT 0,
    min_threshold   INTEGER NOT NULL DEFAULT 0,
    unit_of_measure VARCHAR(20) NOT NULL DEFAULT 'units',
    expiration_date DATE,
    stock_status    VARCHAR(20) NOT NULL DEFAULT 'OK',
    expiration_status VARCHAR(20) NOT NULL DEFAULT 'OK',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_item_sku_branch UNIQUE (branch_id, sku)
) PARTITION BY LIST (branch_id);

CREATE TABLE inventory_deltas (
    delta_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    item_id         UUID NOT NULL,
    branch_id       UUID NOT NULL,
    delta_type      VARCHAR(20) NOT NULL,
    quantity_change  INTEGER,
    absolute_quantity INTEGER,
    reason          TEXT,
    source_ref      VARCHAR(200),
    staff_id        UUID NOT NULL,
    idempotency_key VARCHAR(64) NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_delta_idempotency UNIQUE (idempotency_key)
) PARTITION BY LIST (branch_id);
```

### Materialization Trigger (transactional)

```sql
CREATE OR REPLACE FUNCTION materialize_inventory_delta()
RETURNS TRIGGER AS $$
DECLARE
    new_stock INTEGER;
    item_threshold INTEGER;
    new_status VARCHAR(20);
BEGIN
    -- Lock the item row for concurrency serialization (D-041)
    SELECT current_stock, min_threshold INTO new_stock, item_threshold
    FROM inventory_items WHERE item_id = NEW.item_id FOR UPDATE;

    -- Apply delta
    CASE NEW.delta_type
        WHEN 'INCREMENT' THEN new_stock := new_stock + NEW.quantity_change;
        WHEN 'DECREMENT' THEN
            new_stock := new_stock - NEW.quantity_change;
            IF new_stock < 0 THEN
                RAISE EXCEPTION 'Insufficient stock' USING ERRCODE = 'P0001';
            END IF;
        WHEN 'ADJUST' THEN new_stock := NEW.absolute_quantity;
        WHEN 'THRESHOLD' THEN item_threshold := NEW.absolute_quantity;
        WHEN 'EXPIRATION' THEN NULL; -- handled below
    END CASE;

    -- Compute stock status
    new_status := CASE
        WHEN new_stock = 0 THEN 'OUT_OF_STOCK'
        WHEN new_stock < item_threshold THEN 'LOW_STOCK'
        ELSE 'OK'
    END;

    -- Update materialized view
    UPDATE inventory_items SET
        current_stock = new_stock,
        min_threshold = item_threshold,
        stock_status = new_status,
        updated_at = now()
    WHERE item_id = NEW.item_id;

    -- Notify listeners (< 8KB)
    PERFORM pg_notify('inventory_changes', json_build_object(
        'branchId', NEW.branch_id,
        'itemId', NEW.item_id,
        'deltaType', NEW.delta_type,
        'newStock', new_stock,
        'stockStatus', new_status,
        'timestamp', extract(epoch from now())
    )::text);

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## Permission Model

Existing V010 seeds `inventory:read` and `inventory:manage`. The ADD spec requires finer granularity:

| New Permission | Replaces/Extends | Granted To |
|---|---|---|
| `inventory:read_all` | Extends `inventory:read` for cross-branch | Director General, General Administrator |
| `inventory:read_service` | Scoped `inventory:read` | Service Manager, Physician, Resident |
| `inventory:adjust` | Subset of `inventory:manage` | General Administrator |

New migration V016 will add these 3 permissions. Existing `inventory:read` and `inventory:manage` remain (used by other features in Phase 5).

## WebSocket Security (3 layers)

1. **CONNECT:** JWT validated via `WebSocketSecurityInterceptor`. Invalid/expired token → STOMP ERROR, disconnect.
2. **SUBSCRIBE:** `/topic/branch/{branchId}/inventory` — `branchId` must match JWT `activeBranchId` OR user holds `inventory:read_all`. `/topic/admin/inventory` requires `inventory:read_all`.
3. **RLS:** Even if subscription auth fails, PG RLS blocks data at query level.

## Acceptance Criteria

### MUST
- AC1: 5 delta command types accepted via REST with correct HTTP semantics (201 for mutations, 200 for threshold/expiration)
- AC2: Idempotency — duplicate key returns original result, no second write
- AC3: Stock materialized within same transaction as delta (no eventual consistency)
- AC4: pg_notify → STOMP push delivers change to subscribed client < 2 seconds
- AC5: RLS: service manager query returns only own service items; admin cross-branch works via admin_reporting
- AC6: Partition pruning confirmed via EXPLAIN ANALYZE on single-branch query

### SHOULD
- AC7: LowStockAlert and ExpirationAlert projections correct
- AC8: PWA InventoryRealtimeManager subscribes/unsubscribes on branch switch

### MUST NOT
- AC9: No direct UPDATE on inventory_deltas (append-only enforced)
- AC10: No WebSocket subscription to another branch's channel without inventory:read_all
