# Retrospective — S4.5: Scalability Validation

**Story:** S4.5 | **Size:** S | **Type:** validation
**Started:** 2026-04-13 | **Completed:** 2026-04-13
**Estimated:** 45 min | **Actual:** ~50 min | **Velocity:** 0.9x

## Summary

Validated ESC-02: multi-branch scalability from 3→15 branches with <10% query degradation. All query latencies actually improved at 15 branches (warm DB caches). Partition pruning confirmed via EXPLAIN ANALYZE. Patient search sub-1s at 50K records. WebSocket load test deferred (STOMP framing complexity over k6 raw WebSocket) — architecture-validated through existing unit/integration tests.

## What Went Well

- **k6 scripts worked first try** (after auth fix) — setup/scenario/threshold pattern is clean
- **Partition pruning test in Testcontainers** — fast feedback, deterministic, no Docker Compose dependency
- **Results exceeded expectations** — no degradation at all, queries improved with warm caches
- **Seed script is reusable** — parameterized, portable, covers all major tables

## What to Improve

- **Docker Compose image staleness** not anticipated — wasted ~10 min rebuilding. For infra-dependent stories, add a "verify stack" preflight step.
- **psql meta-command portability** — learned the hard way that `\gset` and `:'var'` don't work through docker exec pipes. Now captured as PAT-A-001.
- **WebSocket STOMP testing** — k6's raw WebSocket module doesn't natively handle STOMP framing (CONNECT/SUBSCRIBE/MESSAGE). A dedicated STOMP k6 extension or a separate tool (Artillery, custom script) would be needed for formal WS load testing.

## Heutagogical Checkpoint

1. **Learned:** psql meta-commands are client-only; template-based SQL is the portable pattern. JWT sharing under concurrent load is a race condition. Append-only triggers affect test cleanup strategy.
2. **Process change:** Add infra-preflight task for stories that depend on Docker Compose or external tooling.
3. **Framework improvement:** Seed script template pattern (sed substitution) should be the standard for all test data scripts.
4. **New capability:** k6 load testing patterns — setup phase token generation, custom metrics, threshold assertions. Directly applicable to future performance validation stories.

## Patterns

- **PAT-A-001** (new): SQL seed scripts must use template placeholders, not psql meta-commands
- **PAT-A-002** (new): Pre-generate per-branch tokens for concurrent load tests

## Discoveries

- Inventory query latencies are flat or improving at 15 branches — partition pruning is highly effective
- PostgreSQL shared_buffers warm-up has more impact on query latency than branch count
- The 50K patient search completes in ~13ms p95 — pg_trgm GIN index is extremely effective
