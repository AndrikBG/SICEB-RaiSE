# Design — S4.4: PWA Views

## Problem & Value

**Problem:** Backend APIs for branch management, inventory CQRS, and tariff configuration exist but have no PWA interface — users can't operate branches without Postman or direct API calls.

**Value:** Completes the E2E user journey: login → select branch → manage inventory → configure tariffs → see real-time updates. Validates M3 (E2E Integration) and proves all three backend stories (S4.1, S4.2, S4.3) work together through a real UI.

## Complexity

Complex: 6 components (4 views + branch switching enhancement + router), 3 backend API integrations, WebSocket real-time, permission gating, IndexedDB cache clearing on branch switch.

## Approach

Build four new views following existing patterns (UserManagementView, RoleConfigurationView): `useState` + `useEffect` for data loading, direct API client calls, Tailwind CSS for styling, `RoleAwareRenderer`/`RequireAnyPermission` for permission gating. No state management library for view state — keep it local.

Two new API clients (`branch-api.ts`, `tariff-api.ts`) following `inventory-api.ts` pattern: typed request/response interfaces, thin wrappers around `api` from `api-client.ts`.

**Branch context switching:** The existing `BranchSelector` in `layout.tsx` only sets `activeBranch` in zustand — it doesn't call `POST /session/branch` to get a new JWT. Enhancement: `BranchSelector.onSelect` → call branch API → update JWT via `updateToken` → clear IndexedDB cache for old branch → reconnect WebSocket (to pick up new JWT in `beforeConnect`) → inventory-realtime auto-resubscribes via existing store watcher.

**Real-time inventory:** `InventoryDashboardView` subscribes via `subscribeInventory()` on mount, updates in-memory grid state when events arrive. No new WebSocket infrastructure needed.

## Examples

### API Client: branch-api.ts

```typescript
import { api } from './api-client';

export interface BranchResponse {
  branchId: string;
  name: string;
  address: string;
  phone: string | null;
  email: string | null;
  branchCode: string | null;
  status: 'ACTIVE' | 'INACTIVE';
  createdAt: string;
}

export interface RegisterBranchRequest {
  name: string;
  address: string;
  phone?: string;
  email?: string;
  branchCode?: string;
}

export interface OnboardingStep {
  stepName: string;
  status: 'PENDING' | 'COMPLETED' | 'FAILED';
  completedAt: string | null;
}

export interface OnboardingStatus {
  branchId: string;
  steps: OnboardingStep[];
  overallStatus: string;
  completedCount: number;
  totalSteps: number;
}

export interface BranchSwitchResponse {
  accessToken: string;
  branchId: string;
  branchName: string;
}

export const branchApi = {
  list: () => api.get<BranchResponse[]>('/api/branches'),
  get: (id: string) => api.get<BranchResponse>(`/api/branches/${id}`),
  register: (data: RegisterBranchRequest) =>
    api.post<BranchResponse>('/api/branches', data),
  deactivate: (id: string) =>
    api.post<void>(`/api/branches/${id}/deactivate`),
  getOnboarding: (id: string) =>
    api.get<OnboardingStatus>(`/api/branches/${id}/onboarding`),
  switchBranch: (branchId: string) =>
    api.post<BranchSwitchResponse>('/api/session/branch', { branchId }),
};
```

### API Client: tariff-api.ts

```typescript
import { api } from './api-client';

export interface TariffResponse {
  tariffId: string;
  serviceId: string;
  branchId: string;
  basePrice: number;
  effectiveFrom: string;
  createdBy: string;
  createdAt: string;
}

export interface CreateTariffRequest {
  serviceId: string;
  basePrice: number;
  effectiveFrom: string;
}

export interface UpdateTariffRequest {
  basePrice: number;
  effectiveFrom: string;
}

export interface TariffPageResponse {
  content: TariffResponse[];
  page: number;
  size: number;
  totalElements: number;
  totalPages: number;
}

export interface TariffSearchResult {
  tariffId: string;
  serviceId: string;
  serviceName: string;
  basePrice: number;
  effectiveFrom: string;
}

export const tariffApi = {
  create: (data: CreateTariffRequest) =>
    api.post<TariffResponse>('/api/tariffs', data),
  update: (tariffId: string, data: UpdateTariffRequest) =>
    api.put<TariffResponse>(`/api/tariffs/${tariffId}`, data),
  getActive: (serviceId: string) =>
    api.get<TariffResponse>('/api/tariffs/active', { params: { serviceId } }),
  list: (params?: { serviceId?: string; includeHistorical?: boolean; page?: number; size?: number }) =>
    api.get<TariffPageResponse>('/api/tariffs', { params }),
  search: (query: string, page?: number, size?: number) =>
    api.get<TariffSearchResult[]>('/api/tariffs/search', { params: { query, page, size } }),
};
```

### Branch Switching Flow (enhanced BranchSelector)

```typescript
async function handleBranchSwitch(branch: BranchInfo) {
  const oldBranchId = activeBranch?.id;
  try {
    const res = await branchApi.switchBranch(branch.id);
    // 1. Store new JWT
    updateToken(res.data.accessToken);
    // 2. Update active branch in store
    setActiveBranch(branch);
    // 3. Clear old branch's IndexedDB cache
    if (oldBranchId) {
      await scopedCachedEntities(oldBranchId).delete();
    }
    // 4. Reconnect WebSocket to pick up new JWT
    disconnectWs();
    connectWs();
    // 5. inventory-realtime auto-resubscribes via auth store watcher
    // 6. Navigate to dashboard
    navigate('/', { replace: true });
  } catch {
    setError('Error al cambiar de sucursal');
  }
}
```

### InventoryDashboardView: Real-time update

```typescript
// On mount: subscribe to real-time updates
useEffect(() => {
  if (!activeBranch) return;
  subscribeInventory(activeBranch.id, (event) => {
    // Update the specific item in the grid without full reload
    setItems(prev => prev.map(item =>
      item.itemId === event.itemId
        ? { ...item, currentStock: event.newStock, stockStatus: event.stockStatus }
        : item
    ));
  });
  return () => unsubscribeInventory();
}, [activeBranch?.id]);
```

### Error: Negative tariff price

```
User enters basePrice = -10.00
→ Client-side validation blocks submit (zod schema)
→ If bypassed, server returns 400: { "code": "SICEB-1001", "message": "Base price must be non-negative" }
→ Error displayed inline in form
```

### Error: Branch switch fails (user not assigned)

```
POST /api/session/branch { branchId: "..." }
→ 403 Forbidden
→ BranchSelector shows "No tienes acceso a esta sucursal"
→ Active branch unchanged
```

## Acceptance Criteria

### MUST

1. `BranchManagementView` at `/branches`: list branches with status, register new, deactivate — gated by `branch:manage`
2. `InventoryDashboardView` at `/inventory`: paginated grid (50/page), filter by status/category/service, search by name/SKU, color-coded status, export Excel — gated by `inventory:read`. Real-time WebSocket updates reflect in grid < 2s.
3. `ServiceInventoryView` at `/inventory/service`: same as dashboard but pre-filtered to user's service — gated by `inventory:read_service`. Items from other services never rendered.
4. `TariffConfigurationView` at `/tariffs`: catalog with create/update/search, non-negative price validation — gated by `tariff:manage`
5. Branch switching calls `POST /session/branch`, stores new JWT, clears IndexedDB cache, reconnects WebSocket

### SHOULD

1. Onboarding progress display in BranchManagementView (step-by-step indicators)
2. Tariff history view (includeHistorical toggle)

### MUST NOT

1. Never render items from other services in ServiceInventoryView (not merely hidden — not fetched)
2. Never show views without required permission (RequireAnyPermission gate, not just hidden links)
3. Never lose the active session on branch switch failure (rollback to previous state)

## Decisions

| # | Choice | Rationale | Alternatives |
|---|--------|-----------|-------------|
| D1 | New `features/operations/` directory for all 4 views | Logical grouping separate from `auth/`, `admin/`, `clinical/` | Put in `features/admin/` (but inventory isn't admin-only) |
| D2 | Local useState for view state, no new zustand store | Views are independent, no cross-view state sharing needed | New `inventory-store.ts` (unnecessary complexity) |
| D3 | BranchSelector enhanced in-place in layout.tsx | Single-branch users auto-select (existing), multi-branch get full switch flow | Separate BranchSwitchModal (over-engineered) |
| D4 | ServiceInventoryView as separate component, not a mode of InventoryDashboardView | Different permissions, different data source (filtered server-side), never shows cross-service items | Single view with mode prop (leaks cross-service data risk) |
| D5 | zod for client-side form validation | Already in dependencies, consistent with react-hook-form | Manual validation (error-prone) |
